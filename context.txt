# RaidenShogun Music Bot — Context

Last updated: 2026-06-27 (session: stop tracking .env, delete dead code, named startup stages, .env.example added)

## Project Overview

A Telegram voice-chat music bot. Group members run /play and the bot's
assistant userbot joins the voice chat and streams audio resolved from
YouTube / Spotify / Resso / SoundCloud via yt-dlp + py-tgcalls + ffmpeg.

There is a second feature set bolted on around the music core: welcome
cards (image-composited), savage leave messages with per-chat anti-repeat,
admin /ban /unban, /id resolver, /toss, /greetings on|off toggle.

## Repository Structure

- main.py                       — process entry point. Calls bot.start.start_bot().
- bot/start.py                  — async runner: loads plugins, starts userbot,
                                  starts PyTgCalls, starts bot, idle().
- bot/client.py                 — Pyrogram Client `app` (bot) and `userbot` (MTProto session).
- bot/config.py                 — loads API_ID/HASH/BOT_TOKEN/STRING_SESSION from .env.
- bot/core/loader.py            — imports every bot/plugins/<x>.py at startup.
- bot/core/music.py             — DEAD FILE. Just `from pytgcalls import PyTgCalls`. Not used.
- bot/utils/music.py            — the real `music = PyTgCalls(userbot)` instance.
- bot/utils/player.py           — yt-dlp wrapper with multi-client fallback chain
                                  (tv_embedded → mediaconnect_frontend → android → web → ios).
                                  Supports COOKIES_FILE env var.
- bot/utils/resolver.py         — turns text/URL into (audio_stream_url, label).
                                  Routes YT/SC to yt-dlp, Spotify/Resso to API then YT search.
- bot/utils/queue.py            — Per-chat playback queue. `Track` dataclass +
                                  now_playing / upcoming / enqueue / pop_next /
                                  clear / is_active / set_current. Lock-guarded.
- bot/utils/chats.py            — Persistent registry of chat IDs the bot
                                  has seen (used by /broadcast since bots
                                  can't enumerate dialogs over MTProto).
                                  remember / forget / all_chats / count.
                                  JSON at $CHATS_FILE (default chats.json).
- bot/utils/owner.py            — Owner identification. Defaults to
                                  userbot.me.id (the user whose STRING_SESSION
                                  is in .env). Override with OWNER_ID env var.
                                  is_owner / get_owner_id, cached.
- bot/utils/downloader.py       — yt-dlp wrappers for /song and /video.
                                  download_audio (mp3@192k via ffmpeg
                                  postprocess) and download_video (≤720p mp4).
                                  Reuses player.PLAYER_CLIENTS for the same
                                  anti-bot-wall fallback. Hard caps: 20 min
                                  duration, 1.5 GB filesize.
- bot/utils/playback.py         — Orchestrator. `play_track(chat_id, track)`
                                  primes userbot peer cache, builds MediaStream
                                  (audio or video), calls music.play. Registers
                                  @music.on_update() to auto-advance queue on
                                  StreamEnded.
- bot/utils/spotify.py          — Spotify API (client-credentials) lookup.
- bot/utils/resso.py            — Resso share-page scraper.
- bot/utils/youtube.py          — older / alt YT helper (need to confirm vs player.py).
- bot/utils/welcome_image.py    — PIL banner+avatar compositor for welcome cards.
- bot/utils/leave_messages.py   — list of leave-line templates + per-chat ring buffer.
- bot/utils/greetings.py        — per-chat on/off flag persisted to greetings.json.
- bot/plugins/*.py              — one file per /command. See "Known Bugs" — most music
                                  plugins are stubs.
- ptb_main.py                   — DELETED 2026-06-27. Was a 830-line python-telegram-bot
                                  parallel implementation. Never wired up; pyrogram path is canonical.
                                  Reference handlers already ported (ensure_userbot_in_chat etc.).
- scripts/install.sh            — VPS one-shot: apt deps + venv + systemd unit pointing
                                  at main.py. Service name: raiden-musicbot.
- requirements.txt              — kurigram==2.2.21, py-tgcalls==2.3.3, TgCrypto==1.2.5,
                                  python-dotenv==1.2.2, yt-dlp>=2026.6.9, aiohttp==3.14.1,
                                  Pillow==12.2.0. (kurigram is a pyrogram fork; code still
                                  does `from pyrogram import ...`.)
- .env                          — UNTRACKED as of 2026-06-27 (was re-added to tracking
                                  after the earlier untrack-commit got reverted).
                                  .gitignore now matches it. History-purge still pending.
- .env.example                  — committed template for new deployers.

## Current Features (what actually works in the deployed pyrogram path)

- /start, /help, /ping                  — static replies + buttons.
- /id                                   — resolve user via reply / @username / id / text-mention.
                                          Uses userbot to look up @usernames bot API can't.
- /ban, /unban                          — admin-gated, supports reply / @username / id /
                                          text-mention. Real chat actions via app.ban_chat_member.
- /toss                                 — coin flip.
- /greetings on|off                     — admin toggle persisted to greetings.json.
- Welcome card on member join           — composited banner+avatar+name+caption.
- Savage farewell on member leave       — per-chat anti-repeat ring buffer.
- Both legacy `new_chat_members` and modern `chat_member_updated` paths.
- /play <text|link>                     — resolves and starts a stream via PyTgCalls.
                                          Also accepts replied audio/voice/video uploads.

### 2026-06-25 — Owner gating, /departure toggle, assistant anti-misuse

User-reported issues addressed in this pass:

1. `/sudolist /addsudo /removesudo /broadcast /stats` rejected the
   deployer ("only for sudo users") even though they thought they were
   the owner.
2. The departure (member-leave) handler was silent because it was gated
   by the same on/off flag as the welcome handler — turning greetings
   off also turned farewells off.
3. The assistant userbot lingered in groups after playback ended, so it
   could be misused.
4. /play in the pyrogram path had no _ensure_userbot_in_chat — the
   userbot couldn't auto-rejoin after item (3) removed it.

Root cause for (1): `bot/utils/owner.py` fell back to `userbot.get_me().id`
(the assistant account's ID, not the deployer's) when `OWNER_ID` env was
unset. The deployer's `.env` had no `OWNER_ID` so the bot considered the
assistant the owner. Additionally `_env_owner()` only accepted a single
digit string (`123,456` failed silently).

Files added:
- bot/utils/departure.py — default-ON per-chat flag, stores the OFF set
  in $DEPARTURE_OFF_FILE (default ./departure_off.json). Atomic write.
- bot/plugins/departure.py — `/departure on|off` admin command. Also
  responds to `/farewell` and `/departures`.

Files rewritten:
- bot/utils/owner.py
  - Accepts comma- or whitespace-separated OWNER_ID env var.
  - New `get_owner_ids() -> set[int]` (back-compat `get_owner_id` kept).
  - New `SUDO_USERS` env var: comma-list pushed into the persistent sudo
    store on first sudo check.
  - Empty-set returned from `get_owner_ids` is NOT cached, so a later
    successful userbot lookup can repopulate it.
- bot/plugins/stop.py — uses new `playback.end_session()` helper.

Files edited:
- bot/plugins/sudo.py — denial messages now include the calling user's
  id and the configured owner ids, so the user can immediately see who
  the bot considers an owner.
- bot/plugins/broadcast.py, bot/plugins/stats.py — same diagnostic
  denial messages, ParseMode.HTML.
- bot/plugins/welcome.py — leave handlers now check
  `departure.is_enabled`; join handlers still check `greetings.is_enabled`.
- bot/plugins/skip.py — when queue empty, calls `end_session` instead
  of `music.leave_call` alone.
- bot/plugins/play.py — before play_track, calls
  `ensure_userbot_in_chat(client, chat_id)` so the assistant rejoins
  after a previous session forced it out.
- bot/plugins/help.py — Welcome & Greetings section now lists
  `/greetings on|off` (welcomes) and `/departure on|off` (farewells)
  separately.
- bot/utils/playback.py — adds:
  * `end_session(chat_id)` — clears queue, leaves the call,
    `userbot.leave_chat(chat_id)`. Best-effort, never raises.
  * `ensure_userbot_in_chat(client_app, chat_id)` — probes
    `get_chat_member(me)`, falls back to
    `app.export_chat_invite_link` + `userbot.join_chat`.
  * `_on_pytgcalls_update` (StreamEnded path) now calls `end_session`
    instead of bare `leave_call` — natural stream-end now also kicks
    the assistant out of the group.
- bot/core/loader.py — added `departure`, `sudo`, `stats` to the
  explicit PLUGINS list. (pyrofork's auto-loader would pick them up
  anyway, but the list is the canonical inventory.)

Behaviour notes:
- Default for departure messages is ON. Existing groups will start
  seeing farewells immediately on first run; admins can `/departure off`.
- `/greetings` no longer affects farewells. Existing chats that had
  greetings ON will continue to receive welcomes; chats with greetings
  off but who DO want farewells should `/departure on` (no-op since
  default-ON) and leave greetings off.
- Assistant auto-leave fires on: natural stream-end with empty queue,
  `/stop`, `/end`, `/skip` when the next pop is empty. Manual operator
  action (kicking the assistant from VC mid-stream) is not caught here

- For `/play` to auto-invite the assistant back, the BOT must be a
  group admin with invite-link rights. Otherwise the user has to
  re-invite the assistant manually each session. The play handler
  reports a clear error.

### 2026-06-24 — /broadcast (owner-only fan-out)

Added `/broadcast` for the bot owner to push a message to every chat the
bot has been seen in.

Two forms:
- `/broadcast <text>` — sends text to every chat
- Reply to a message with `/broadcast` — copies the replied message
  verbatim (text, media, captioned media — `Message.copy` preserves all)

In groups/supergroups the resulting message is pinned silently. DMs are
not pinned by design (intrusive).

Files added:
- bot/utils/chats.py (chat registry, JSON-persisted, lock-guarded,
  atomic save via tmp+rename)
- bot/utils/owner.py (derives owner_id from userbot.get_me() the first
  time, cached; OWNER_ID env var overrides)
- bot/plugins/broadcast.py (the command + a passive group=-1 handler
  that records chat_ids from every incoming message)

Files edited:
- bot/core/loader.py (added "broadcast" to PLUGINS list)

Behaviour notes:
- Owner gate: rejects non-owners with 🔒.
- FloodWait: caught, slept for the requested seconds + 1, retried once.
- "Dead chat" errors (PeerIdInvalid, UserIsBlocked, ChatWriteForbidden,
  ChannelPrivate, ChannelInvalid, UserIsBot) drop the chat from the
  registry so future broadcasts skip it.
- 0.05s pacing between sends (~20/s) so we don't trip the per-bot limit.
- Returns a 4-line summary: sent / pinned / failed / forgotten.

Chat tracking caveat: bots only learn about chats from incoming messages,
so the first broadcast right after deploy will reach only chats that have
exchanged a message with the bot since the registry was created. There's
no way to backfill (Telegram doesn't expose dialog enumeration to bots).

Verification:
- py_compile of all new/changed files: PASS
- Unit-tested chats.remember/forget/all_chats: PASS
- Bot restarted cleanly with the new plugin loaded.
